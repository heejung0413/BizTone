## Problem Statement

실무자가 작성한 Raw 텍스트(개조식, 불완전한 문장)를 비즈니스 상황에 맞는 메일로 변환하고 싶다. ChatGPT에 매번 "이거 메일로 바꿔줘, 상사에게 보내는 거야" 같은 설명을 하는 것이 번거롭다.

## What Makes This Cool

**프리셋 톤** — 매번 톤을 설명하지 않아도 됨. "상사 보고", "동료 요청", "외부 업체" 중 하나만 선택하면 한국 비즈니스 문화에 맞는 메일이 생성됨. ChatGPT보다 빠르고, 기존 톤 변환 도구들(HyperWrite, Mailmodo)보다 한국어/한국 직장 문화에 특화됨.

## Constraints

1. MVP는 웹앱 + 메일 변환 1개(3가지 프리셋)만 구현
2. LLM: Claude API (claude-3-5-sonnet)
3. Frontend: React + Vite + Tailwind CSS
4. Backend: Node.js (Express) 분리
5. 배포: Frontend(Vercel) + Backend(Railway 또는 Render)
6. v1에서 제외: Chat/Casual/Report 포맷, 히스토리, 톤 조절, 역할 설정, 크롬 익스텐션

## Premises

1. MVP는 '웹앱 + 메일 변환'만 — 동의함
2. Claude API 사용 — 동의함 (한국어 자연스러움 우선)
3. React + Vite + Tailwind — 동의함
4. 백엔드 분리 (API 키 보안) — 동의함

## Approaches Considered

### Approach A: Minimal MVP (S)
- 입력창 + "메일로 변환" 버튼 1개 + 출력창
- Effort: S (1-2일)
- Pros: 가장 빠름
- Cons: ChatGPT와 차별점 거의 없음

### Approach B: Preset 기반 (M) ⭐ 선택됨
- 한국 비즈니스 맥락 프리셋 3개
- Effort: M (3-5일)
- Pros: 핵심 차별점 구현, 한국 직장 문화 반영
- Cons: 프리셋별 프롬프트 튜닝 필요

### Approach C: 프롬프트 엔지니어링 집중 (M)
- Few-shot 예시 + Role prompting으로 품질 최대화
- Effort: M (3-5일)
- Pros: 변환 품질 향상
- Cons: 프롬프트 튜닝에 시간 많이 소요

## Recommended Approach

**Approach B: Preset 기반** 선택.

사용자가 "프리셋 톤 - 매번 설명 안 해도 됨"을 핵심 가치로 꼽았고, 한국 비즈니스 맥락 프리셋이 기존 영어 중심 도구들과의 차별점이 될 수 있음.

### 프리셋 정의

| 프리셋 (UI) | API Key | 상황 | 톤 특징 |
|-------------|---------|------|---------|
| 상사 보고 | `boss` | 팀장/임원에게 보고 | 격식체, 결론 먼저, 간결, "드립니다" 어미 |
| 동료 요청 | `colleague` | 같은 팀/타팀 동료에게 요청 | 반말X 존댓말, 부드럽지만 명확, 부탁 표현 |
| 외부 업체 | `external` | 협력사/고객사에게 | 비즈니스 격식, 회사 대 회사, 정중함 |

### 기술 구조

```
[Frontend - Vite + React + Tailwind]
├── src/
│   ├── App.tsx              # 메인 레이아웃
│   ├── components/

│   │   ├── PresetButtons.tsx # 프리셋 선택 버튼 3개
│   │   └── ResultDisplay.tsx # 결과 표시 + 복사 버튼
│   └── api/
│       └── convert.ts       # 백엔드 API 호출
├── .env                     # VITE_API_URL
└── package.json

[Backend - Express]
├── src/
│   ├── index.ts             # Express 서버
│   ├── routes/
│   │   └── convert.ts       # POST /api/convert
│   └── prompts/
│       └── email.ts         # 프리셋별 프롬프트 정의
├── .env                     # ANTHROPIC_API_KEY
└── package.json
```

### API 설계

```
POST /api/convert
Request:
{
  "text": "회의 2시에서 3시로 변경됨. 장소는 회의실A 그대로",
  "preset": "boss" | "colleague" | "external"
}

Response (성공):
{
  "result": "안녕하세요, 팀장님.\n\n오늘 예정된 회의 시간이 변경되었습니다.\n- 기존: 오후 2시\n- 변경: 오후 3시\n- 장소: 회의실A (변경 없음)\n\n일정 조정에 참고 부탁드립니다.\n감사합니다."
}

Response (에러):
{
  "error": "에러 메시지",
  "code": "INVALID_INPUT" | "EMPTY_TEXT" | "API_ERROR" | "RATE_LIMIT"
}
```

**입력 검증:**
- `text`가 비어있으면 → `EMPTY_TEXT`
- `text`가 5000자 초과 → `INVALID_INPUT`
- `preset`이 유효하지 않으면 → `INVALID_INPUT`
- Claude API 호출 실패 → `API_ERROR`

## Success Criteria

- [ ] 입력 텍스트를 3가지 프리셋 중 하나로 메일 변환 가능
- [ ] 변환 결과 복사 기능 동작
- [ ] Vercel + Railway 배포 완료
- [ ] 본인이 실제로 업무에서 1주일 사용

## Distribution Plan

- Frontend: Vercel (자동 배포, main 브랜치 연동)
- Backend: Railway 또는 Render (환경변수로 API 키 관리)
- 도메인: Vercel 기본 도메인으로 시작, 필요시 커스텀 도메인

## What I noticed about how you think

- "단순한 웹 앱으로 충분"이라고 하셨는데, 이건 좋은 판단이에요. 크롬 익스텐션까지 한 번에 만들려다 완성 못하는 프로젝트가 많습니다.
- "프리셋 톤 - 매번 설명 안 해도 됨"이라고 정확히 짚으셨어요. 이게 ChatGPT 직접 사용과의 핵심 차이점입니다.
- 백엔드 분리가 필요하다고 직접 지적하셨는데, 프론트엔드 개발자가 보안 이슈를 인식하고 있다는 건 좋은 신호입니다.
- MVP 범위를 "웹앱 + 포맷 1개만"으로 명확히 줄이신 것도 좋아요. 기획서에는 기능이 많았는데 실행 가능한 범위로 스스로 조정하셨습니다.
